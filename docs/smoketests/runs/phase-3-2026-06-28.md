## Finding F1 — Phase 4 pre-empted Phase 3's timing AC

**Analysis.** The PRD §6 Phase 3 AC named "sponsor `/login` mobile Speed Index drops from 3.67 s toward 0.85 s (admin-login parity)" with the 3.67 s figure sourced from `recon/perf_investigation_2026_06_18.md` §Finding #5. That recon ran against a sponsor app that still served 428 KB of Unsplash hot-linked imagery on `/login` (Phase 4 scope). After Phase 4's imagery strip landed on `main`, sponsor `/login` Speed Index dropped to ~0.8 s on its own. Phase 3's preload relocation contributes ~5 ms on top of that — sub-noise against the ~5–10% single-run Lighthouse variance.

The recon's root-cause attribution over-credited the preload: the preload IS waste, but its timing impact on Speed Index alone is small. The 3.67 s → 0.85 s headline came from the imagery + preload combined, not the preload in isolation.

**Methodology caveat.** This analysis isolates Phase 3 against the post-Phase-4 baseline only. The full causal chain (preload alone on the pre-Phase-4 imagery-bearing baseline) is not measured here — that would require running Lighthouse against a third state (pre-Phase-4 main with Phase 3 stashed) which is unavailable now that Phase 4 has merged. The "Phase 4 pre-empted Phase 3's timing AC" framing is the strongest claim the available data supports; a stricter framing would be "on the current post-Phase-4 baseline, Phase 3's Speed Index contribution is sub-noise — whether the preload alone would have driven the recon's 3.67 s observation is no longer separately verifiable."

**Decision.** Keep Phase 3 shipping — the routing-contract win remains load-bearing (eliminates wasted server work + DB roundtrip on every login page load, and adds a Playwright regression guard against future placement bugs). Re-frame Phase 3's primary AC from "Speed Index drop" to "Playwright routing contract." Soften the Speed Index AC to "directionally improves vs the same-build baseline" with a ±200 ms tolerance for single-run variance.
