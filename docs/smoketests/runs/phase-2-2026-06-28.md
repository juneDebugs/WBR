## Empirical finding — Next.js 15.5.15 auto-injects viewport meta defaults

Surfaced while capturing the Step 4 BASELINE measurement. Recorded per the established sprint finding protocol; Phase 2 itself is non-breaking and ships unchanged. Verification of the iOS implication is batched into the UAT round.

### Observation

The pre-Phase-2 sponsor app source — `apps/sponsor/app/layout.tsx` with `export const viewport = { themeColor: '#6366f1' }` only — produces served HTML containing `<meta name="viewport" content="width=device-width, initial-scale=1"/>` when built via `pnpm --filter sponsor build` on Next.js 15.5.15. Verified via two paths: (a) `curl http://localhost:3003/login` against the locally-served baseline build, and (b) direct inspection of the pre-built `apps/sponsor/.next/server/app/login.html` file.

### Pattern (triangulated)

Next.js 15.5.15 starts with `width=device-width, initial-scale=1` as the framework default and merges the `viewport` export's properties on top. Empirical evidence:

| App / state | Viewport export source | Served `<meta name="viewport">` content |
|---|---|---|
| Sponsor pre-Phase-2 | `{ themeColor: '#6366f1' }` | `width=device-width, initial-scale=1` |
| Sponsor post-Phase-2 | `{ width: 'device-width', initialScale: 1, themeColor: '#6366f1' }` | `width=device-width, initial-scale=1` (identical) |
| Attendee pre-Phase-2 (from stale build cache) | `{ width: 'device-width', initialScale: 1, maximumScale: 1, userScalable: false, themeColor: '#6366f1', viewportFit: 'cover' }` | `width=device-width, initial-scale=1, maximum-scale=1, viewport-fit=cover, user-scalable=no` |
| Attendee post-Phase-2 | `{ width: 'device-width', initialScale: 1, themeColor: '#6366f1', viewportFit: 'cover' }` | `width=device-width, initial-scale=1, viewport-fit=cover` |

### Implication for Phase 2

The Phase 2 sponsor edit (adding `width: 'device-width', initialScale: 1` to the viewport export) is functionally a no-op at the rendered-HTML level on Next.js 15.5.15. The served meta tag is identical pre- and post-edit. The PRD §6 Phase 2 stated cause-and-effect — "viewport export currently only sets themeColor; iOS 15+ renders sponsor pages at 980px desktop width" — does not hold empirically on the current framework version.

Three possible explanations for the original 2026-06-18 Finding #2 diagnosis (`recon/perf_investigation_2026_06_18.md` §Finding #2):

1. The original diagnosis was a code-read inference, not a real-device observation. The reader did not account for Next.js's auto-injection of viewport defaults.
2. Next.js's auto-inject behavior was added in a version released between 2026-06-18 and 2026-06-28. The behavior on the version used during recon may have differed.
3. There is a real iOS layout problem on sponsor pages, but its root cause is not the viewport meta tag — Phase 2's sponsor edit does not fix it.

The Phase 2 attendee + meetings edits (removing `userScalable: false` and `maximumScale: 1`) remain substantive — the served HTML changes materially, dropping the zoom-locking attributes. Real a11y win, real impact on users with low vision.

### Sponsor edit value, restated

Even though the sponsor edit is a no-op at the served-HTML level on this Next.js version, the change has non-zero engineering value:

- Documents intent in source: a future reader sees explicit `width`, `initialScale`, `themeColor` values and knows they're intentional, not accidents of framework defaults.
- Guards against framework default-behavior drift: if a future Next.js upgrade changes auto-inject behavior, the explicit values protect the contract.
- Consistency with attendee + meetings: all three apps now follow the same explicit-viewport-export pattern.
