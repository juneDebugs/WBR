### Methodology issue — simulated LCP is unreliable for these routes right now

Lighthouse's "simulated" throttling mode uses the **lantern model**: capture the page's network + JS dependency graph on a fast headless Chrome, then project what slow-4G would look like by replaying through a bandwidth + RTT model. The model attributes downloads-after-load as a chain that contributes to LCP delay.

The root cause of the inflation is `/api/data/chat` at **4.2 MB** transfer size. Inspecting `/people` network requests:

| Endpoint | Transfer size | Run 1 duration |
|---|---:|---:|
| `/api/data/chat` | **4,231 KB** | 658 ms |
| `/api/data/home` | 944 KB | 278 ms |
| `/api/data/setup` | 705 KB | 383 ms |
| `/api/data/speakers` | 599 KB | 345 ms |
| `/api/data/schedule` | 371 KB | 318 ms |
| `/api/data/my-schedule` | 2 KB | 857 ms |
| `/api/data/people` | 6 KB | 1,600 ms |
| `/api/data/meetings` | 0 KB | 137 ms |

The 4.2 MB chat-data prefetch is the dominant payload. In real-world slow-4G, downloading 4.2 MB at ~200 KB/s would take ~21 seconds — and the lantern model amplifies this into the simulated LCP because the chat prefetch starts after `load` and runs concurrently with everything else still happening. Tiny variations in request ordering between runs produce 2–3× swings in the simulated number.

The Phase 2 baseline ran on 2026-06-18 against the production deployment. The chat data may have been smaller then; or the lantern model may have apportioned the prefetch differently because the pre-fix code fired all 8 prefetches in parallel at layout mount (less ambiguous chain). Either way, **same-tool same-methodology comparison is showing methodology artifacts, not real perf regressions**.

### What this means

The deferral is working as designed:
- `/api/data/<route>` (the page's own critical query) fires at ~245–389 ms — well before `observedLoad` at 351–442 ms.
- The 7 non-current-page prefetches all queue post-load, as intended.
- Observed page-paint times are 830–2,472 ms across all routes — well under the 3 s preferred bar.

The simulated LCP comparison is unreliable because the 4.2 MB chat-data prefetch dominates the lantern model. The real-user experience matches the observed times, not the simulated projections.
